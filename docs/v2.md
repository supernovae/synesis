# Synesis v2 — Architecture Simplification Roadmap

Captured from an architecture audit (March 2026) comparing Synesis's compound AI
pipeline against frontier monolithic models (Claude, GPT-4, Gemini).

The current architecture is **valid for 8B–30B models** where orchestration
compensates for model size. As models improve or larger models become feasible,
progressively simplify.

---

## 1. How Frontier Models Differ

| Aspect | Frontier (400B+) | Synesis Today (8B–30B) |
|--------|-------------------|------------------------|
| Intent classification | Native — model infers from user message | Entry classifier + supervisor (two nodes) |
| Tone / depth | Native — model adapts to topic automatically | Taxonomy YAML config shapes system prompt |
| Safety / quality | Baked in via RLHF / Constitutional AI | Runtime critic node (extra inference pass) |
| Output format | Direct markdown streaming | JSON wrapper `{"code":"...","explanation":"..."}` parsed by respond node |
| Tool use | Model decides when to call tools | External graph routes to sandbox/RAG/search |

Frontier models handle all of this in **one inference pass**. The multi-node
graph exists to give smaller models the same effective capability.

---

## 2. Simplifications to Implement (ordered by impact)

### 2a. Skip Critic for Low-Complexity Explain-Only

**Why:** A full 8B inference round-trip to review "5 creative things for kids
art" adds latency with almost no quality gain.

**How:** In `graph.py` routing, when `deliverable_type == "explain_only"` and
`complexity_score < 0.5`, route directly from `worker → respond`, bypassing
`patch_integrity_gate` and `critic`.

**Risk:** Low. Non-code explain_only queries have no sandbox, no patch ops.
The critic adds little value here.

### 2b. Direct Markdown Streaming for Explain-Only

**Why:** The JSON wrapper `{"code":"...","explanation":"..."}` creates:
- Parsing failure modes (the invalid `\escape` bug)
- Time-to-first-visible-token delay (must parse JSON before extracting content)
- Complexity in `StreamingCodeExtractor`

**How:** For explain-only deliverables, skip the JSON wrapper. System prompt
becomes: "Respond in markdown." Worker streams markdown directly. No JSON
parsing, no `validate_with_repair`, no extraction. The respond node passes
through as-is.

**Risk:** Medium. Requires a separate code path for explain-only vs. code
generation. The JSON format is needed for code (patch_ops, files_touched, etc.)
but not for documents/explanations.

### 2c. Adaptive Node Skipping

**Why:** The full graph has ~10 nodes. Simple queries traverse most as no-ops
(0ms each) but still incur async overhead, logging, and state serialization.

**How:** Add a `fast_path` flag. When entry classifier determines
`task_size == "trivial"` and `output_type == "document"`, skip:
- `strategic_advisor` (already 0ms for non-code)
- `supervisor` (already bypassed for trivial)
- `planner` (not needed for explain-only)
- `context_curator` (no RAG for trivial)
- `patch_integrity_gate` (no code to check)
- `sandbox` (no execution)
- `critic` (see 2a)

Direct path: `entry_classifier → worker → respond` (3 nodes instead of 10).

**Risk:** Low for trivial document queries. Must not trigger for code.

### 2d. Model-Size-Aware Taxonomy Tuning

**Why:** As you move to 70B+ models (or LoRA-tuned 30B), the model will
naturally infer appropriate depth and tone. The taxonomy config becomes
redundant overhead.

**How:** Add a `model_capability_tier` setting (small/medium/large). When
`large`, taxonomy prompt shaping is disabled — depth_instructions and
worker_explain_tone are not injected. The model handles it natively.

**Risk:** Low. The YAML config already supports this gracefully (empty
`depth_instructions` = no injection).

## 3. Relevant Research and Frameworks

### Compound AI Systems (Berkeley AI Research, 2024)
The formal framework for what Synesis is: composing multiple models, retrievers,
and tools into a pipeline. Validates the architecture for smaller models.
Key paper: "The Shift from Models to Compound AI Systems" (Zaharia et al.)

### RouteLLM (LMSys, 2024)
Routing queries to different-sized models based on complexity. Similar to the
entry classifier + supervisor pattern. Could replace the supervisor with a
lightweight router that picks model size, not just route.

### DSPy (Stanford NLP, 2024)
Programmatic optimization of LLM pipelines. The taxonomy YAML config is a
manual version of what DSPy automates. Consider DSPy for auto-tuning prompt
templates and few-shot examples per domain.

### Skeleton-of-Thought (Microsoft Research, 2023)
Generate outline first, then fill in parallel. Could apply to complex
explain-only queries: planner generates outline, worker fills sections in
parallel batches.

### Speculative Decoding
Use a small model to draft, large model to verify. The worker + critic pattern
is conceptually similar. Could be formalized: use 8B for first draft, 30B
critic only when confidence < threshold.

## 4. UX Patterns to Watch (2026 Expectations)

- **Persistent user preferences**: "I prefer concise answers" should persist
  across sessions. Memory partially handles this; consider explicit preference
  storage.
- **Adaptive verbosity**: Track user feedback signals (thumbs up/down, "make
  it shorter", follow-up depth requests) to auto-tune response length over time.
- **Confidence signaling**: When the model is uncertain, say so. Frontier
  models do this naturally; smaller models need prompting or calibration.
- **Progressive disclosure**: Stream a summary first, offer "tell me more"
  expansion. Reduces perceived latency for complex queries.
- **Conversation-aware routing**: Second message in a thread should inherit
  the first message's routing decisions, not re-classify from scratch.

## 5. When to Simplify

| Trigger | Action |
|---------|--------|
| Move to 70B+ model | Disable taxonomy prompt shaping (2d) |
| Explain-only latency > 3s | Implement direct markdown streaming (2b) |
| Simple query latency > 1s | Implement fast-path node skipping (2c) |
| Critic adds < 5% quality lift on non-code | Skip critic for explain-only (2a) |
| LoRA fine-tuning deployed | Reduce orchestration; model handles more natively |

---

## 6. What's Sound Today

The current plan (config-driven prompt shaping via `taxonomy_prompt_config.yaml`)
is the **right next step** because:
1. Zero inference cost (deterministic YAML lookup)
2. Meaningfully improves 8B–30B output quality for diverse topics
3. Config-driven = easy to disable when models improve
4. Doesn't add latency or failure modes
5. Aligns with Compound AI Systems research

Implement the prompt-shaping plan first. Revisit this roadmap after measuring
latency and quality on the deployed system.
